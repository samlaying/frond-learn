# 02 - 三层架构练习

## 目标

把一个真实业务功能拆成 UI 层、业务实体层、DB / 通讯层。

## 业务背景

这章不再用“抽象按钮”来练习，而是用一个更接近生产环境的场景：

- 客服在工单系统里处理用户投诉
- 需要查看工单、编辑回复草稿、做校验、保存到后端
- 页面里会有 loading、成功、失败、模板填充、慢网模拟这些状态

这个场景更接近真实项目，因为它天然会逼出分层需求：

- 页面展示和输入属于 UI
- 是否允许保存、回复是否超长属于业务规则
- 保存草稿、后端返回、网络延迟属于通讯层

对应的练习入口在：

- [三层架构版本](/Users/sam/03-Code/frond-learn/practice/frontend-architecture/02-layered-architecture/layered.html)

## 练习任务

1. 选一个功能场景，比如“发送消息”或“保存草稿”。
2. 列出这个功能里哪些代码应该放 UI 层。
3. 列出哪些代码应该放业务实体层。
4. 列出哪些代码应该放 DB / 通讯层。
5. 把一个最小版本的目录结构写出来。

## 对比理解

### 没有分层时，生产环境里常见的样子

通常会是一个页面文件里，直接把这些东西写在一起：

- 读取输入框内容
- 校验长度和空值
- 拼请求参数
- 调接口
- 改 loading 状态
- 改按钮文案
- 显示成功 / 失败结果

这种写法短期能跑，但问题也很现实：

- 一次点击事件会越来越长
- 接口改了，要去大块脚本里找
- 规则变了，UI 也容易被一起改坏
- 很难把同一套规则复用到别的页面

### 分层后，生产环境里的常见样子

真实团队里通常会拆成几块：

- `ui`：收集输入、展示状态、改页面文案
- `domain`：处理校验、构建业务 payload、决定能不能提交
- `repository`：负责调用后端接口或本地存储
- `main`：把三层串起来

这样做的好处是：

- 改 UI 不会碰接口细节
- 改校验规则只动业务层
- 后端接口换字段，只改通讯层
- 复杂业务更容易测试

### 一个更贴近真实项目的目录长什么样

```text
02-layered-architecture/
  index.html
  no-layer.html
  layered.html
  layered/
    main.js
    ui.js
    domain.js
    repository.js
    state.js
```

这已经是很典型的“前端生产结构”了：

- 页面负责入口
- `main.js` 负责组装
- 业务规则独立出去
- 通讯层单独封装

## 怎么记住这几层

你不用一开始就背“UI / domain / repository”这几个英文名，先记中文就够了。

### 最好记的口诀

> 看得见的放 UI，算规则的放业务，碰接口的放数据。

这个口诀基本能覆盖大多数前端业务。

### 你可以再缩成三个问题

每次看到一段代码，先问自己：

1. 这是给人看的，还是给机器用的？
2. 这是在做判断，还是在改页面？
3. 这是在算规则，还是在发请求？

如果是给人看的，通常进 UI。
如果是在做判断，通常进业务层。
如果是在发请求、读写存储，通常进通讯层。

### 你以后判断 AI 代码对不对，也可以按这个看

拿到 AI 生成的代码，不要先纠结代码写得漂不漂亮，先看它有没有把这三件事混在一起：

- UI 里有没有直接写校验规则
- 业务层里有没有直接操作 DOM
- 通讯层里有没有知道页面按钮、弹窗、文案

如果混了，结构就不对。

### 一个简单的自检方法

你可以给自己留一个固定检查顺序：

1. 先看有没有明显的 UI 代码
2. 再看有没有明显的业务判断
3. 再看有没有明显的请求 / 存储
4. 最后看它们是不是被分到不同文件里

只要这四步能过，你就大概率知道它是不是按层拆了。

### 你还可以记住一个边界

> 一层只回答一种问题。

- UI 回答“怎么展示”
- 业务层回答“该不该做”
- 通讯层回答“从哪取、往哪存”

## 检查点

- UI 层是否只负责交互和展示
- 业务实体层是否不依赖 UI
- DB / 通讯层是否只负责数据存取和传输
- 一个功能变更时，是否能只改对应的一层
- 复杂状态是否不会全部堆在一个文件里

前端应用可以拆成 UI 层、业务实体层和基础设施层：

```text
UI 层（JSX、用户操作、外观呈现）
        ↓
业务实体层（业务对象与业务规则）
        ↓
DB / 通讯层（IndexedDB、WebSocket）

辅助能力：Event Bus、Web Worker、PWA Service Worker
```
# 一张图讲清前端架构 — 详细笔记

> **来源：** Bilibili / CTO王丹尼 | 2026-02-27
> **项目案例：** 畅聊（ChangLiao）— 面向中国用户的社交媒体平台，支持即时通讯、新闻浏览、自动切换服务器

---

## 一、基础前提：单页架构（SPA）

单页架构出来之前，每次访问网站都要重新加载 HTML、CSS、业务数据和业务逻辑，访问量大，速度慢。

**单页架构的核心思想：**
- HTML/CSS 等结构文件在首次加载时一次性获取
- 后续用户操作（如点击按钮看新闻）只从后台取数据
- 大大提升了前端应用的性能（performance）

---

## 二、架构总览：三层架构 + 辅助层

整个前端项目被分为 **三个核心层次（Layer）**，95% 以上的代码都应放在这三层之中：

```
┌─────────────────────────────────────────┐
│              UI 层（控件层）               │
│        JSX / 用户操作 / 外观呈现           │
├─────────────────────────────────────────┤
│           业务实体层（Business层）          │
│      所有业务逻辑 / 纯 JavaScript          │
├──────────┬──────────┬───────────────────┤
│  DB 层   │  通讯层   │  PWA Service Worker │
│ (IndexedDB)│(WebSocket)│   (离线/安装体验)    │
├──────────┴──────────┴───────────────────┤
│              Web Worker                   │
│          (繁重计算任务)                     │
└─────────────────────────────────────────┘
```

---

## 三、八大核心架构原则

### 原则 1：严格分层

95%以上的代码必须放在 UI 层、业务实体层、DB/通讯层这三层之一。

### 原则 2：单向调用（从上往下）

- UI 层 → 业务实体层 → DB/通讯层
- **不能反向调用**
- 仅上面的层依赖下面的层，这样可以减少约 50% 的代码复杂度

### 原则 3：Event Bus（消息总线）处理下层向上通信

当下层确实需要通知上层时，使用 **Event Bus / 消息驱动模式**，而非直接调用。

### 原则 4：UI 层必须很薄

- 只负责：用户操作 + 外观呈现（look and feel）
- **绝对不能包含业务逻辑**

### 原则 5：业务实体层是重中之重

- 程序的核心应该放在业务实体层
- 业务实体层 **不知道** UI 层的存在

### 原则 6：前端可以有数据库 + 独立通讯层

- DB 层和通讯层是独立的
- 这些层 **不知道** 上层业务实体的存在

### 原则 7：繁重计算任务放 Web Worker

- 视频压缩、转码等 CPU 密集型任务放到 Web Worker
- 避免阻塞主线程，保证 UI 流畅

### 原则 8：PWA Service Worker 实现 App 级体验

- 像开发普通网站一样开发
- 交付给用户的却是 App 级体验（离线推送、桌面安装等）

## 四、分层原理详解（附案例）

### 4.1 为什么要分层？

**核心目的：降低代码复杂度，减少 30%~50% 的代码量。**

### 4.2 案例：畅聊即时通讯功能

**UI 组件：**
- 聊天室（Chat Room）
- 聊天历史对话框（Chat History）
- 输入条（Input Bar）：含输入框、发送按钮、文件选择、按住说话

**最差的设计：** 把发送逻辑写在发送按钮的 `onClick` handler 里面 → UI 代码和业务逻辑搅在一起。

**正确的做法：**

输入条的 `keyDown` 处理函数非常简单：
1. 拿到字符
2. 如果按了回车 → 调用发送函数
3. 发送函数几行代码就结束

**关键点：**
- 调用的都是业务实体层的对象（如 ChatRoom、ChatItem）
- UI 层只做很轻很薄的事，真正的逻辑全部交给业务层

### 4.3 分层带来的迁移优势

**实际经历：**
- 畅聊最初只做「私聊」页面
- 半年到一年后，需要新增「公共讨论区」页面

**如果当初没有分层：** 把聊天逻辑从私聊 UI 迁移到公共讨论 UI → 像做脑外科手术，至少大半个月。

**因为严格分层：**
- 聊天室的业务逻辑（Room、Participant、Chat History、通讯渠道、数据库存储）全部独立于 UI
- 新的公共讨论页面代码 **几行就搞定**
- 迁移连一天都没用到，所有时间都用在开发新功能

> **如果只能选一个最重要的原则，就是：分层、分层、分层！**

## 五、层间调用关系详解

### 5.1 从上往下的调用

- 私聊 UI 和公共讨论 UI 都直接调用业务层的 `Room`
- `Room` 完全不知道上面是谁在调用，也不关心
- 同样，业务层调用通讯层（WebSocket）也很简单，只需一个 `sendMessage`

### 5.2 通讯层的复杂性自行消化

畅聊需要对抗恶劣的网络环境，通讯层内部要处理：

| 场景 | 通讯层的处理 |
|------|------------|
| 服务器被封 | 自动切换到其他服务器 |
| WiFi → 5G 切换 | 网络断开后自动恢复连接 |
| 数据丢失 | 自动重新获取数据 |

业务层只需一个指令下去，**不需要关心通讯层出什么问题**。

### 5.3 从下往上的调用：Event Bus 消息总线

**场景：** 我发送了消息，接收方的聊天室怎么知道有新消息来了？

**不能直接调用上层** → 使用 **Event Bus 设计模式**。

## 六、Event Bus / 消息驱动模式

### 6.1 核心思想

- 用一个数据结构在各模块之间传递消息
- 一个模块发送/广播，另一个模块接收并处理
- 消息在各模块间 **几乎原封不动地传递**，每个模块拿到消息后才执行自己的业务逻辑

### 6.2 畅聊中的实现（第一版 → 优化版）

#### 第一版实现：

**通讯层（WebSocket）的 `onMessage`：**
1. 收到 WebSocket 消息
2. 根据消息类型分发（Switch 语句）
3. 如果是聊天消息（`P2P_SEND_MESSAGE`）：
   - 通过 `RoomManager` 找到目标聊天室
   - 直接调用 `Room.processData()`

**聊天室（Room）的 `processData`：**
1. 处理业务逻辑
2. 调用 `window.dispatchEvent()` —— 系统函数，将事件发送到浏览器的全局消息总线

**UI 控件层：**
1. 在控件创建之初，用 `addEventListener` 监听事件
2. 收到消息后做简单处理：检查 room ID 是否合法 → 更新页面

**第一版的问题：**
通讯层混入了业务代码（`RoomManager`、`findRoom`、`processData`），耦合度偏高。

#### 优化版实现：

- 将 `RoomManager` 消息分发逻辑搬回 `RoomManager` 本身
- `RoomManager` 有自己的消息处理函数
- 通讯层的代码简化为 **一行**：直接把消息传给 `RoomManager`
- 通讯层彻底不包含任何业务逻辑

### 6.3 设计模式总结

> **消息驱动（Event Bus）** 是一种被广泛应用的设计模式。
> Windows 操作系统中窗口间协作用的就是完全相同的机制：
> 鼠标点击 → OS 收集事件 → 系统消息总线 → 分发到目标窗口 → 窗口内处理

## 七、前端数据库：IndexedDB

- 所有浏览器都自带数据库 —— **IndexedDB**
- 适合大量数据缓存（如视频文件）
- `localStorage` / `sessionStorage` 只适合存很少的内容
- IndexedDB 是键值对型数据库，非 SQL 型
- 对前端应用非常有用

---

## 八、Web Worker：提升性能的利器

### 8.1 问题：浏览器的单线程瓶颈

- 浏览器默认一个网页只用一个 CPU 内核
- 所有任务排队处理（队列式）
- 视频压缩需要几分钟 → 用户要干等几分钟，所有按钮失效

### 8.2 解决方案：Web Worker

- 为繁重任务另开一个线程，在另一个 CPU 上运行
- **完全不影响主线程的 UI 处理**

### 8.3 畅聊的实际案例

上传 800 多兆视频 → 切成多个小段 → 每段独立压缩 → 分发到各服务器

**在压缩过程中：**
- 用户可以正常发送消息
- 可以接收别人发的消息和图片
- UI 完全不受影响

### 8.4 Web Worker 使用方法

**两个核心概念：**
1. Web Worker 生活在另一个线程中
2. 主线程和 Worker 之间 **只能通过消息通信**，不能直接调用

**步骤：**

**① 创建 Worker：**
```javascript
const worker = new Worker('worker-file-url', {
  type: 'module'  // 跟 JS 版本有关
});
```

**② 主线程接收 Worker 消息：**
```javascript
worker.onmessage = function(event) {
  // 处理从 Worker 返回的消息
  processMessage(event.data);
};
```

**③ 主线程向 Worker 发消息：**
```javascript
worker.postMessage({
  type: '消息类型ID',
  data: '消息内容'
});
```

**④ Worker 内部接收消息：**
```javascript
self.onmessage = function(event) {
  const msg = event.data;
  // 根据 msg.type 分发处理
};
```

**⑤ Worker 内部向主线程发消息：**
```javascript
self.postMessage({
  type: '进度/结果',
  data: result
});
```

**消息处理方式：** 和通讯层一样，根据消息类型 Switch 分发处理。

## 九、PWA：跨时代的前端技术

### 9.1 PWA 赋予网站 App 级体验的三个方面

| 特性 | 说明 |
|------|------|
| **桌面安装** | 安装为手机/电脑桌面上的图标，点击即启动，无需输网址 |
| **离线操作** | 关掉屏幕后仍能收到推送消息（类似微信） |
| **离线可用** | 所有代码已缓存到本地，服务器不可用时仍可正常运行 |

### 9.2 推送通知的平台差异

| 平台/情况 | 支持情况 |
|-----------|---------|
| **苹果手机 + 海外** | 完全支持 |
| **苹果手机 + 大陆** | 支持 |
| **安卓 + 海外** | 支持（依赖谷歌推送系统） |
| **安卓 + 大陆** | 不支持浏览器推送 |
| **QQ浏览器 / UC浏览器** | 部分支持 |
| **百度浏览器 / 搜狗浏览器** | 完全不支持（连安装都不行） |

### 9.3 PWA 对畅聊的战略意义

畅聊需要对抗恶劣的网络环境：
1. 主服务器被封 → PWA 已缓存全部代码，下次启动可自动升级
2. 自动寻找其他可用服务器继续运行
3. 正常看新闻、聊天不受影响

## 十、核心要点速查表

| # | 原则/技术 | 一句话总结 |
|---|----------|-----------|
| 1 | 分层 | UI / 业务实体 / DB+通讯，三层各司其职 |
| 2 | 单向调用 | 从上往下调用，减少 50% 复杂度 |
| 3 | Event Bus | 下层向上通知用消息总线 |
| 4 | 薄 UI | UI 层只管外观和用户操作 |
| 5 | 业务实体层是核心 | 程序重心在此，且不知道 UI 的存在 |
| 6 | 前端 DB + 独立通讯层 | IndexedDB 存大数据，通讯层自管复杂网络 |
| 7 | Web Worker | 繁重计算放独立线程，不卡 UI |
| 8 | PWA | 网站开发，App 体验，离线可用 |

## 十一、关键收获

1. **分层是最重要的架构决策** — 直接决定了代码量和可维护性
2. **消息驱动模式是复杂应用的必备设计模式** — 操作系统也用同一套思路
3. **Web Worker 是提升前端性能的「便宜又好用」的方案** — 立竿见影
4. **PWA 是前端的跨时代技术** — 让网站具备 App 的离线、推送、安装能力
5. 架构的价值不在于理论，在于面对真实复杂需求时（如新增页面、对抗恶劣网络）带来的实际收益

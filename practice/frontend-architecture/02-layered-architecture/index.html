把“客服工单草稿”拆成真正能落地的生产结构

这不是教学玩具，而是一个接近真实业务的页面：客服要处理用户消息、编辑回复草稿、调用接口保存、查看历史记录。 你会看到两种实现方式，一种是没有分层的写法，一种是按 UI / 业务 / 通讯 分层后的写法。

看没有分层的版本 看三层架构版本 看教程说明

业务背景

客服在工单系统里处理一条用户消息，边看内容边写回复草稿，还要保存到后端，方便别的同事接手。

真实压力

会有加载中、失败重试、输入校验、草稿自动保存、状态回填这些问题，代码不可能永远只有一个按钮。

练习重点

不是把页面切成三块，而是看清楚：哪些逻辑属于展示，哪些属于规则，哪些属于请求和存储。

没有分层时的常见样子

  • 按钮点击里同时写校验、请求、DOM 更新。
  • 接口地址、业务规则、提示文案混在一个文件里。
  • 改一个输入校验，容易顺手把页面逻辑改坏。
  • 后面想换接口、换 UI、加自动保存，会越来越乱。

分层后的生产写法

  • UI 只负责收集输入、展示状态。
  • 业务层只负责规则判断和状态流转。
  • 通讯层只负责调接口、读写本地数据。
  • 后续加功能时，可以只改一层，不会全局乱动。
建议你先打开“没有分层的版本”，再打开“分层版本”，对照看同一个业务在两个结构里怎么组织。

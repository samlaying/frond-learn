把“调用方向”练成一条单向链

这章不是让你背概念，而是让你知道：一个功能怎么从 UI 一路走到业务层，再走到通讯层。 重点不是“谁更高级”，而是“谁只负责哪一段”。

这页会把你刚才那四句话直接落到一个真实流程里。你可以把它当成“输入消息并发送”的练习板： 先选一个流程，再画调用链，然后故意试一次坏设计，最后改成只允许上层调用下层。

业务流程：输入消息并发送

你可以把它当成聊天框、工单回复框，或者留言提交框。
1. UI：用户输入消息
↓
2. 业务层：校验消息、组装 payload
↓
3. 通讯层：请求接口发送消息
↓
4. UI：展示成功 / 失败状态
这就是你要记的单向链路： UI -> 业务实体 -> 通讯层。 你只要盯住这一条方向，就不会把代码写成互相跳来跳去。
对应你那四步，可以直接这么理解：
  • 先挑一个具体业务流程，不要上来就想整个系统。
  • 把“谁调用谁”画成一条链，而不是互相乱依赖。
  • 可以故意写一次反向调用，再删掉，看看哪里会乱。
  • 最后把结构改成：上层只依赖下层，不反过来依赖。

运行日志

这里会记录谁调用了谁。你观察它，就能理解“单向调用”是什么意思。
判断标准很简单：
如果你看到“通讯层去改 UI”或者“业务层去碰按钮”，那就是方向错了。
